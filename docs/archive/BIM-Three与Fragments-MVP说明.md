# BIM Three.js + Fragments MVP 说明

更新时间：2026-06-24

## 1. 文档目的

本文是当前 `Three.js + Fragments` BIM Viewer MVP 的唯一执行说明，合并原有需求整理、阶段进度记录、任务清单和历史 WebIFC 迁移方案中的有效信息。

当前有效技术路线为：

```text
IFC -> Fragments .frag -> Three.js + Fragments 前端渲染
```

当前验证项目位置：

```text
E:\xeokit-bim-viewer\fragments-convert-poc
```

当前页面入口：

```text
http://127.0.0.1:5175/viewer/index.html
```

启动命令：

```bat
cd /d E:\xeokit-bim-viewer\fragments-convert-poc
npm run serve
```

## 2. 背景与当前决策

前期曾验证过 `xeokit-convert -> XKT` 和 `Three.js + web-ifc` 路线。实测中，`web-ifc` 直接打开和加载几何的速度可以接受，但 `xeokit-convert` 在大模型元数据、属性集解析和 XKT 生成链路上存在明显耗时瓶颈。

因此当前不再把 `xeokit + XKT` 或纯 `Three.js + web-ifc` 作为主线，而是优先验证 Fragments：

- 使用 `.frag` 作为浏览器端 BIM 模型格式。
- 使用 `@thatopen/fragments` 负责模型加载、构件拾取、可见性、颜色、透明度、属性读取等核心能力。
- 使用 Three.js 负责渲染、相机、控制器和常规三维交互。
- 后端生产级 IFC/RVT 转换服务暂不在当前 MVP 中完成，只验证前端 Viewer 能力是否可支撑业务。

当前锁定版本建议：

```json
{
  "@thatopen/fragments": "3.4.5",
  "three": "0.182.0",
  "web-ifc": "0.0.77"
}
```

## 3. MVP 验证范围

当前阶段目标不是完成完整 BIM 平台，而是验证 Fragments 是否能支撑以下 Viewer 基础能力：

- `.frag` 模型稳定加载和浏览。
- 模型树展示。
- 构件点击选择。
- 构件属性、材质和基础信息展示。
- 构件隐藏、显示、隔离、着色、透明度、定位。
- 基础视角切换、全屏、快照、视点保存和恢复。
- 框选工具模式。
- 右键菜单。
- 大模型下模型树、拾取、属性读取和构件操作是否保持可用性能。

当前阶段暂不实现：

- RVT 直接解析。
- RVT 自动转换 IFC。
- 生产级 IFC 转 `.frag` 转换服务。
- Java 调用 Node.js 转换服务封装。
- 多用户协同、权限、审批、批注流转。
- 云端对象存储、断点续传、任务队列。
- 图纸和三维模型的精准坐标配准。

当前已处理的问题：

- `classes` 树在大模型下不再一次性展开全部构件叶子，避免 DOM 过多和栈溢出。
- `classes` 改为类型聚合节点。

- IFC 类型名会尝试转为更可读名称，例如 `IFCWALL -> Wall`。

## 6. 当前已知限制

属性完整度取决于 `.frag` 转换阶段是否写入完整属性和关系。如果转换时未保留 `Name / Description / PSet / 材质 / 楼层关系 / 空间关系`，前端无法凭空恢复这些 BIM 语义数据。

`objects` 树目前仍是普通 DOM 树，大模型下可能继续需要：

- 懒加载节点。
- 虚拟滚动。
- 默认只展开少量层级。
- 点击节点时再生成子节点 DOM。

当前视点只保存在浏览器内存中，刷新页面后丢失。后续需要持久化到本地配置或业务数据库。

框选和右键菜单已具备基础形态，但仍需要真实大模型验证准确性和性能。

## 7. P0 任务：稳定当前 MVP

### T-001 大模型加载性能验证

目标：

- 使用 100MB 级 IFC 转换后的 `.frag` 文件进行加载验证。
- 记录首屏加载时间、可交互时间、内存占用、旋转缩放流畅度。

建议记录字段：

```text
模型名称
原始 IFC 大小
.frag 文件大小
加载耗时
构件数量
浏览器内存
是否卡顿
是否崩溃
```

验收标准：

- 模型能稳定加载并显示。
- 页面不长时间无响应。
- 旋转、缩放、平移可用。
- 日志能记录加载耗时和构件数量。

### T-002 模型树性能验证与优化

目标：

- 验证 `objects / classes / storeys` 在大模型下的生成速度和页面性能。
- 避免一次性渲染过多 DOM 节点导致页面卡顿。

后续任务：

- 验证 `objects` 树在大模型下是否仍然卡顿。
- 增加树节点懒加载。
- 增加树搜索。
- 增加树节点虚拟滚动。
- 控制默认展开层级，例如只展开到 `Project / Site / Building / Storey`。

验收标准：

- 大模型加载后左侧树不导致页面明显卡顿。
- 展开、折叠、点击节点响应可接受。
- `classes` 不再出现 `Maximum call stack size exceeded`。

### T-003 属性完整度验证

目标：

- 验证 `.frag` 是否保留足够的 BIM 语义数据。
- 明确默认转换和完整属性转换之间的差异。

需要验证字段：

```text
GlobalId
Name
LongName
Description
ObjectType
PredefinedType
Pset_*
材质
楼层关系
空间关系
分类信息
```

需要对比两种转换方式：

```js
// 默认转换
await importer.process(...)

// 完整属性转换
importer.addAllAttributes()
importer.addAllRelations()
await importer.process(...)
```

验收标准：

- 构件点击后能稳定显示 `localId` 和 `GlobalId`。
- 常见构件能显示真实名称或类型。
- 能判断完整属性转换是否值得进入后续方案。

### T-004 构件 ID 稳定性验证

目标：

- 判断同一 IFC 文件重复转换为 `.frag` 后，业务可依赖的 ID 是否稳定。

验证方式：

1. 同一 IFC 文件使用相同版本、相同参数转换两次。
2. 加载两个 `.frag` 文件。
3. 对比同一构件的 `GlobalId`、`localId` 和模型树结构。

验收标准：

- IFC `GlobalId` 必须稳定。
- Fragments `localId` 在同版本、同参数、同文件输入下应保持稳定。
- Three.js `object.uuid` 不作为业务主键。

推荐 ID 策略：

```text
业务主键：projectId + modelVersionId + ifcGlobalId
前端交互：modelId + localId
调试对象：Three.js object.uuid
```

### T-005 框选能力验证

目标：

- 验证当前框选工具是否能稳定选择构件。
- 验证大模型下 `rectangleRaycast` 的性能和结果准确性。

验收标准：

- 框选模式和浏览模式不冲突。
- 框选时模型不会被误旋转。
- 框选结果能高亮或显示在右侧面板。
- 大模型下框选耗时可接受。

### T-006 右键菜单准确性验证

目标：

- 验证不同模型、不同视角下右键拾取是否准确。
- 验证右键菜单操作是否和当前命中构件一致。

需要验证操作：

```text
定位
隔离
隐藏
着色
全部显示
```

验收标准：

- 右键命中的构件与鼠标位置一致。
- 菜单操作作用于正确构件。
- 框选模式下右键菜单禁用或不触发冲突。

## 11. 当前主要风险

### 11.1 `.frag` 属性完整度风险

如果转换时没有写入完整属性和关系，前端无法可靠展示 BIM 语义信息。需要对比默认转换和完整属性转换，记录文件体积、转换耗时、加载耗时和属性完整度后再决定生产转换参数。

### 11.2 大模型树性能风险

大模型的 `objects` 树可能包含大量节点，普通 DOM 树会有性能瓶颈。后续需要懒加载、虚拟滚动和默认展开层级控制。

### 11.3 构件选择准确性风险

点击选择、框选、右键拾取都依赖 Fragments 拾取能力和当前相机状态。需要用多个真实模型验证，并在右侧面板展示当前命中集合便于人工核对。

### 11.4 后端转换路线风险

当前 MVP 只验证前端渲染和交互，不代表最终生产转换链路已确定。后续仍需验证：

- Java 如何调用 Node.js Fragments 转换服务。
- 转换服务如何队列化、限流、记录日志。
- 转换失败如何重试和反馈。
- 原始 IFC、生成 `.frag`、元数据如何存储。

## 12. 阶段完成标准

满足以下条件时，可认为 `Three.js + Fragments` MVP 适配性验证通过：

- 100MB 级模型转换后的 `.frag` 能稳定加载。
- 大模型下基础浏览可用。
- 模型树可用且不会明显拖慢页面。
- 构件点击、树节点点击、框选、右键拾取基本准确。
- 右侧面板能显示可用的构件 ID、分类、名称、属性、材质信息。
- 构件隐藏、隔离、透明、着色、定位操作稳定。
- 明确 ID 策略：业务使用 `GlobalId`，前端交互使用 `localId`。
- 明确是否需要完整属性转换参数。

## 13. 历史路线说明

原 `Three.js + web-ifc` 方案用于验证浏览器直接解析 IFC 的可行性，结论是几何加载速度可以接受，但长期 BIM Viewer 能力需要大量自研，包括模型树、构件操作、属性读取、剖切、测量、多模型管理和内存释放等。

当前已将主线切换为 `Three.js + Fragments`，WebIFC 方案不再作为当前执行路线，仅保留其结论：浏览器端 IFC 几何解析可作为技术参考，但当前 MVP 以 `.frag` 为前端模型格式。
