## 1. 文档目的

本文档根据 `BIM-Three与Fragments架构功能扩展可行性评估.md` 中的功能优先级，结合当前 `BimModelThree` 项目实际实现状态，整理后续功能开发计划。

计划目标：

- 明确 P0-P3 阶段的开发顺序。
- 区分已实现、部分实现、待实现功能。
- 给出每个阶段的交付物和验收标准。
- 作为后续迭代、任务拆分和进度跟踪依据。

## 2. 当前项目基础

当前技术路线：

```text
IFC -> Fragments IfcImporter / converter-service -> .frag / manifest.json
-> Three.js + @thatopen/fragments Viewer
-> 模型树 / 属性面板 / 构件交互 / iframe 集成 / SDK facade
```

当前已具备的基础能力：

- `.frag` 和 `manifest.json` 加载。
- Three.js 场景、相机、OrbitControls。
- 模型适配、标准视图切换。
- 构件选择、框选、右键菜单。
- 构件定位、隐藏、隔离、显示全部。
- 构件着色、透明度调整、重置颜色。
- 快照。
- 内存版视点保存和恢复。
- 模型树基础展示、懒加载、分批渲染、详细树弹窗。
- iframe `postMessage` 集成桥。
- Viewer SDK facade。
- converter-service 基础转换、manifest/report 输出、HTTP 任务服务 MVP。

## 3. 总体实施原则

1. 先稳定当前 MVP，再补新功能。
2. 优先做会影响后续功能可信度的基础验证，例如属性完整度、ID 稳定性、拾取和树性能。
3. P1 先补 BIM Viewer 高频基础能力：捕捉、测量、视点持久化、标签、剖切。
4. P2 再进入业务化能力：批注、视图浏览器、分区、多模型和模型装配。
5. P3 的复杂增强能力后置，不与 P0/P1 并行抢资源。
6. 所有需要业务绑定的数据优先使用 `GlobalId`，前端运行时操作继续使用 `localId`。

## 5. P0：稳定当前 MVP

### 5.1 阶段目标

P0 阶段不追求新增复杂功能，重点是把当前 Viewer 的数据可靠性、交互稳定性和大模型性能打牢。

### 5.2 任务拆分

| 任务 | 说明 | 前端工作 | 后端/转换工作 | 验收标准 |
|---|---|---|---|---|
| T003 属性完整度验证 | 验证 `.frag` 是否包含业务所需属性 | 属性面板字段梳理、字段缺失展示 | 对比 IFC 原始属性与转换后属性 | 输出属性完整度报告 |
| T004 ID 稳定性验证 | 验证 `GlobalId/localId` 稳定性 | 前端选择和属性展示绑定验证 | 多次转换对比 ID 映射 | 确认业务主键策略 |
| 大模型树性能优化 | 优化模型树 DOM 和属性读取 | 懒加载、分批渲染、详细树、搜索过滤 | 可选：后端预生成树索引 | 大模型展开不卡死 |
| 框选验证 | 验证 rectangleRaycast 准确性 | 框选交互、耗时日志、结果数量展示 | 无 | 不同模型框选结果稳定 |
| 右键菜单准确性验证 | 验证右键命中和菜单动作 | 命中构件、定位/隐藏/隔离/着色联动 | 无 | 右键命中和操作无错乱 |

### 5.3 当前进展

- 大模型树性能已开始优化：已实现懒加载、分批渲染、详细树弹窗、选中态增量更新。
- 框选已实现，仍需要真实模型验证准确性和耗时。
- 右键菜单已实现初版，仍需要真实模型验证命中准确性。
- 属性完整度和 ID 稳定性已有评估方向，但需要形成正式验证报告。

### 5.4 P0 交付物

- 属性完整度验证报告。
- ID 稳定性验证报告。
- 大模型树性能优化记录。
- 框选和右键菜单真实模型验证记录。
- MVP 基线验收清单。

## 6. P1：补齐基础 BIM Viewer 能力

### 6.1 阶段目标

补齐 BIM Viewer 高频基础能力，使系统从“可查看、可选择”升级到“可测量、可标注、可剖切、可恢复视图”。

### 6.2 任务拆分

| 任务 | 当前状态 | 推荐实现方式 | 前置条件 | 验收标准 |
|---|---|---|---|---|
| 捕捉 | 基础版已实现，待验收 | Fragments `snapRaycast()` 或 raycast 辅助点 | 拾取稳定、单位明确 | 鼠标移动可命中可用捕捉点 |
| 距离测量 | 基础版已实现，待验收 | raycast/snapRaycast + Three.js Line + 文本标签 | 捕捉基础能力 | 点到点距离可展示、可撤销、可清除 |
| 视点持久化 | localStorage 基础版已实现，待验收 | 保存 camera state、target、selection、snapshot | 后端或本地存储结构 | 刷新后可恢复视点 |
| 三维标签 | 基础版已实现，待验收 | CSS2DRenderer / CSS3DRenderer | `GlobalId/localId` 绑定 | 标签可显示、点击可联动构件 |
| 基础剖切 | 基础版已实现，待验收 | Three.js Plane + Fragments 剖切能力 | 模型渲染稳定 | 单剖切面可开启、移动、关闭 |

### 6.3 推荐开发顺序

1. 捕捉。
2. 点到点距离测量。
3. 视点持久化。
4. 三维标签。
5. 基础剖切。

### 6.4 P1 交付物

- 捕捉工具。
- 距离测量工具。

|---|---|---|---|---|

## 9. 阶段时间建议

| 阶段 | 建议周期 | 说明 |

| P0 MVP 稳定化 | 1.5-2.5 周 | 真实模型验证占比较高 |
| P1 基础 BIM 能力补齐 | 3-5 周 | 捕捉、测量、标签、剖切需要反复调交互 |

## 11. 验收口径

每个功能进入“完成”状态前，至少满足：

- 可在真实模型上运行。
- 有明确入口和退出方式。
- 操作过程不会破坏模型选择、显隐和相机状态。
- 大模型下没有明显卡死。
- 关键状态有日志或可观察结果。
- 涉及业务绑定时必须使用 `GlobalId` 或明确记录 `localId` 的运行时限制。

## 12. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| IFC 属性不完整 | 标签、批注、分区不可用 | 先做 T003，必要时调整转换参数或补属性索引 |
| ID 不稳定 | 视点、批注、标签绑定失效 | 先做 T004，业务主键优先使用 `GlobalId` |
| 大模型树卡顿 | Viewer 可用性下降 | 懒加载、分批渲染、搜索过滤、虚拟滚动 |
| 拾取性能不稳定 | 测量、捕捉、右键菜单不可靠 | 做 T005 拾取和框选性能验证 |
| 多模型坐标不一致 | 模型叠加不准确 | 先确认 BIM 原始坐标，再补 transform 调整 |
| 业务范围扩张 | 排期失控 | P0/P1 只做基础能力，复杂协同后置 |

## 13. 结论

当前系统已经具备 BIM Viewer MVP 基线能力，后续不应直接进入复杂业务功能，而应优先完成 P0 稳定化。

推荐路线：

```text
P0 稳定 MVP
-> P1 补齐捕捉、测量、视点、标签、剖切
-> P2 批注、视图浏览器、分区、多模型
-> P3 漫游、关键帧、双视窗、标签聚合、图纸联动
```

其中 P0 和 P1 是近期主线，P2 根据试点反馈选择性推进，P3 建议作为专项预研和增强阶段。
