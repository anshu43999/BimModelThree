## 2. 当前基线

| 项 | 当前结论 |
|---|---|
| Viewer 主格式 | `.frag` |
| 推荐资源入口 | `bim-model-manifest/v1` |
| Viewer 技术栈 | Three.js `0.182.0` + Fragments `3.4.5` |
| 主 Viewer | `viewer-service/viewer/index.html` |
| 功能 Demo | `viewer-service/viewer/demo.html` |
| 第三方推荐方式 | iframe + `BimViewerEmbedClient` |
| 直接 SDK | 静态 ESM 基础版，当前未发布 npm 包 |
| 业务数据 | 文件型 CRUD + SDK adapter；数据库、权限和审计待完成 |
| 自动化结果 | 2026-07-16：`109 passed / 0 failed` |
| 人工验收 | 未完成，必须使用真实项目模型执行 |
| 唯一完全未实现的主要前端专项 | 同源图纸模型联动 |

当前主链路：

```text
IFC -> converter-service -> .frag + manifest.json
    -> viewer-service -> iframe / JavaScript SDK -> business system
```

## 5. 文档权威顺序

同一内容出现冲突时，按以下顺序判断：

1. 接口与第三方调用：第三方 SDK 对接说明 > Manifest/SDK 规范 > 其他方案文档。
2. 当前功能状态：当前现状 > 进度主表 > 功能验收清单 > 可行性评估。
3. 时间、难度和剩余工作量：只以进度主表为准。
4. 验收结论：只以功能验收清单中人工填写并签署的结果为准。
5. Viewer 优化任务：以系统优化方案中的 OPT 状态为准。
6. 归档文档不参与当前结论。

历史需求、旧开发计划、专项规则、部署方案和阶段记录位于 [archive/README.md](archive/README.md)。主目录当前只保留 8 份需要持续维护的文档。

归档原则：

- 被新文档替代的文件移动到 `archive`，不删除。
- 归档内容保留决策追溯价值，不继续维护“当前状态”。
- 重新启用历史方案时，在主目录创建或更新当前文档。

## 7. 维护规则

1. 文档使用 UTF-8 Markdown。
2. 每份当前文档必须包含更新时间和文档状态。
3. 功能代码状态变化时，至少同步当前现状、进度主表和验收清单。
4. SDK 方法或事件变化时，同步 Manifest/SDK 规范、第三方 SDK 手册和契约测试。
5. 计划中的“已实现”只表示代码完成；“已验收”必须有真实模型记录。
6. 不在多份文档重复维护同一组剩余人日。
7. 新文档加入后必须更新本索引；被替代文档必须归档。
